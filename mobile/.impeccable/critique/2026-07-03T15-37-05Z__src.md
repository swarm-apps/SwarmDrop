## Anti-Patterns 判定

**LLM 评审**:这是由懂产品的人搭出来的 UI,不是模板拼装。系统字体是刻意的中性选择、色彩几乎只花在状态上、`active:opacity-70` 反馈全局统一、破坏性动作全有二次确认、错误文案有"友好房东"口吻的翻译层。未满分的原因是几处熟练用户会停顿的微偏差:主屏"在线"绿字对比不足且与分享屏用色不一、卡片圆角漂移、蓝底白 spinner。

**确定性扫描**:detect.mjs 对 src 下 128 个源文件零发现(退出码 0)。引擎经脏样本验证真实生效(构造 `#ff0000` 正确报出),零发现是真干净。手工交叉核对:3 处 hex 均合法(色板内 amber、iOS shadowColor)、无渐变文字、无 bounce/elastic 缓动;321 处 `text-[Npx]` 是 DESIGN.md 刻意的 10-15px 紧凑字阶,正确不报。

**交叉印证**:检测器零发现与 A 的"非模板拼装"结论互相印证。但 A 抓到了检测器原理上抓不到的问题——text-success 作为正文文本使用违反 DESIGN.md 的 State Ink Rule(色值本身在色板内所以检测器不报,错在语义用法)。检测器的视觉层规则(对比度、热区、溢出)因无渲染环境未运行,该空缺与 A 标注的"需真机验证"项重合,留待真机评估补齐。

**视觉 overlay**:跳过——原生 RN 应用,无 web 构建,无已启动模拟器/真机。

## 用户四问的直接回答

**流畅?** 代码层是流畅的:150-160ms FadeIn、active:opacity-70 即时反馈、进度行 memo 化、原生 tab bar、尊重"减弱动效"、列表虚拟化、无编排式加载动画。真实减分项:内容区加载全用居中 spinner 而非骨架屏(Skeleton 原语存在却 14 屏零使用),弱网下内容会"闪 spinner 再 pop"。动画手感、掉帧需真机验证。

**简洁?** 大体简洁:近单色底+状态色、10-15px 紧致字阶、单一边框色、系统性渐进披露。拉低两处:主屏顶部把过多 CTA/指标挤在拇指够不到的上半屏;圆角/spinner 色/标题缩进等 token 未收敛的微噪点。

**不廉价?** 基本不廉价、有质感:刻意的系统字体、无渐变/毛玻璃、错误分类考究、文案有人味。会掉价的正是那几个熟练用户能察觉的 tell(在线绿字对比不足、圆角不齐、蓝底白 spinner)。修掉即从"好"跨到"精致"。

**业务契合?** 强契合,直接落地"The Trusted Doorstep":被动接收优先、状态先于装饰、掌控来自可见性、桌面 token 对齐,5 条设计原则里 4 条执行干净。唯一打折:原则 3(亲和优先于黑话)——"打洞/中继/Peer ID/NAT"泄漏在一级界面。另有一个北极星层面的矛盾:E2E 加密只在引导出现一次,正式界面全程无可见凭证——核心信任主张恰恰不可见。

## 认知负荷

1 个失败 + 1 个临界(整体属低负荷):
- 失败——单一焦点:主屏((main)/index.tsx)节点运行时同时是状态仪表盘+配对控制台+设备目录+传输监视器,顶部可交互元素超过 8 个。
- 临界——设备策略编辑器信任级别 4 项(device/[peerId].tsx:580),踩工作记忆上限,因在详情页+高级项折叠判可接受。
- 渐进披露是系统性长处:诊断/高级策略/更多筛选/添加设备面板全部默认收起。

## 情感旅程

**峰值**:收文件弹窗接收前展示信任徽标+策略+保存位置+完整文件树;传输失败给"重新发送"向前出路;破坏性确认强调可逆。
**低谷**:①E2E 加密全程不可见;②配对接受失败静默;③入站分享被打断即丢失(_layout.tsx:193-204,v1 不暂存)。
**建议**:加低调"端到端加密"凭证行;配对/接收失败补 toast.error;入站分享做一次性暂存。

## 做得好的三点

1. **错误处理成熟度远超同类**:friendlyTransferError(transfer/shared.tsx:259)把无法穷举的 Rust 英文错误启发式映射成暖调中文;收件箱详情区分"瞬时失败给重试"与"确认不存在不给重试"(inbox/[itemId].tsx:252-294)。
2. **渐进披露是系统性的**:一级界面对新手干净,深度留给极客。
3. **破坏性动作两级降权+可逆文案**:传输详情主操作全宽实心、破坏性操作降权窄居中无填充(transfer/[sessionId].tsx:471-573),红色永不撑成 hero。

## 优先问题

- **[P1] 主屏"在线"绿字对比不足且跨屏不一致**——device-card.tsx:76,150 用 text-success(约 3.3:1,低于 AA),share-target.tsx:393 同标签用合规 text-success-ink。全 App 最高频状态指示器,无障碍+一致性双违规。改法:text-success→text-success-ink(顺带 (main)/index.tsx:365 的 text-destructive→text-destructive-ink)。
- **[P2] 内容区用 spinner 而非骨架屏**——Skeleton 原语存在却 14 屏零使用。改法:列表/详情加载态改骨架行。
- **[P2] AppHeader 双重内边距致标题缩进 40px 不与卡片(20px)对齐**——mobile/screen.tsx:78 的 px-5 嵌在 AppScreen px-5 内。改法:去掉 AppHeader 的 px-5。
- **[P2] 卡片圆角语汇不统一**——DESIGN 规定 card=rounded-lg,但 hero 面板/InfoPill/预览/图标 chip 出现 rounded-2xl、rounded-[28px]。改法:收敛或把分级规则写进 DESIGN.md。
- **[P2] 高风险动作失败静默**——pairing-request-host.tsx:60-68、transfer-offer-host.tsx:134 失败仅 console.warn 后关弹窗。改法:dismiss 前补 toast.error。
- **[P3] 主按钮 loading spinner 用白色**——send/select-device.tsx:309、pairing-request-host.tsx:139 在 Trust Blue 上用 colors.background,重现 Unified Ink Rule 修过的低对比;其余 22 处均正确。
- **[P3] 搜索清除按钮 28px 无 hitSlop**——inbox/search.tsx:186(全库 20 处 hitSlop 纪律良好,此为孤立遗漏)。

## Persona red flags

**Casey(分心移动/单手/弱网)**:配对入口在展开面板中段、节点控制(StatusPill)在右上角拇指够不到;入站分享被打断即丢失;弱网无骨架屏反复闪 spinner。好的一面:配对 15s UI 超时兜底、策略草稿后台刷新时保留。

**Jordan(首次使用)**:一级黑话泄漏("打洞/中继"、"Peer ID/NAT"、"临时/协作设备"无内联解释);"收件箱"vs"活动"概念重叠,收到的文件被拆两处,心智割裂。好的一面:引导 3 屏价值优先、节点未启动直接给"启动节点"。

**Wei(隐私极客)**:E2E 加密引导后全程不可见——最在意的核心主张不可见,与"控制来自可见性"矛盾;快速撤销信任约 4-5 步无快捷路径;配对决策点只显截断 peerId,接受瞬间无法核验完整指纹。好的一面:4 级信任语义清晰,"· 待确认"后缀诚实。

## 次要观察

- 两份几乎重复的 StatePill(inbox-list.tsx 与 inbox/[itemId].tsx),DRY 隐患。
- 一次性 row 原语散落(InfoRow/DetailRow/Row/SpecRow/DetailLine/Metric),内边距微差。
- "收件箱"与"活动"多入口加剧主屏顶部 CTA 竞争。
- 引导标题 26/22px 超出字阶规则,但属合法 drenched welcome 例外。
- 全局尊重"减弱动效";除复制外几乎无 haptics,配对成功/传输完成两个峰值可用触觉强化。

## 发人深省的问题

1. 主屏同时是状态仪表盘、配对控制台、设备目录、传输监视器。既然原则 #1 说"被动接收优先、配对才是主动动作",主屏是否应只把配对作为唯一 hero 动作?
2. "控制来自可见性"承诺下,E2E 加密只在引导可见一次。"这是加密的"该住在哪里,才能让 Wei 每次传输看得到,又不打扰 Jordan?
3. 收件箱(收到的内容)和活动(传输过程)是同一批东西的两个家。这个拆分服务的是用户心智模型,还是系统架构?
