## 为什么

SwarmDrop 正在从“传输历史列表”升级为可信数据通道。既然收到的内容会进入真正的收件箱，旧的“历史记录”就不应该继续作为主功能存在，而应该重塑为解释传输过程、失败原因、暂停状态和恢复入口的“活动与恢复”视图。

本变更把内容视角和过程视角拆开：收件箱回答“我收到了什么？”，活动与恢复回答“这次传输发生了什么，还能不能恢复？”。

## 变更内容

- 新增 Drop Inbox，作为接收端查看已完成收件内容的主界面。
- 新增 `inbox_items` 数据模型，引用传输会话，但不重复承载传输过程状态。
- 只有成功接收完成的内容才生成收件箱记录；暂停、中断、失败、取消的传输留在活动与恢复中。
- 将现有传输历史 UI 重塑为活动与恢复：
  - 活跃传输
  - 暂停/可恢复传输
  - 失败/取消诊断记录
  - 已完成传输作为次要过程上下文
- 保留 `transfer_sessions` 和相关传输记录，作为进度、恢复、诊断和审计的过程账本。
- 为收件箱增加打开、显示位置、另存/导出、删除记录、查看传输详情等操作。
- 明确删除语义：
  - 删除收件箱条目只删除内容视角记录，并可选择是否删除本地已接收文件。
  - 清空活动与恢复不能删除收件箱内容。
- v1 仅覆盖接收的文件/目录；文本、剪贴板、Artifact Bundle 先保留数据模型扩展空间，不在本变更实现。

## 能力

### 新增能力

- `drop-inbox`: 面向内容的收件箱，用于成功接收的文件/目录，包含来源元数据、内容操作和保留/删除语义。
- `transfer-activity-recovery`: 面向过程的活动与恢复视图，用于活跃、暂停、失败、取消、已完成和可恢复的传输会话。

## 影响

- **crates/entity + crates/migration**: 新增收件箱条目持久化，以及与现有传输 session/file 记录的关系。
- **crates/core**: 接收会话完成时创建收件箱记录；暴露收件箱查询/操作 API；让收件箱投影和传输活动投影保持分离。
- **src-tauri**: 新增列出收件箱、打开/显示/另存/删除收件箱条目、查询活动与恢复数据的 Tauri commands。
- **src**: 新增收件箱路由/界面，重塑传输历史为活动与恢复，更新导航文案和删除/保留交互。
- **SwarmDrop-RN**: 如果共享 core API 变化，移动端需要后续适配；本变更可先落桌面端。
- **文档**: 更新产品说明，明确收件箱与活动的职责边界。
