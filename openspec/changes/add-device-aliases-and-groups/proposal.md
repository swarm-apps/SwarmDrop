## Why

PeerId、系统 hostname 和对端自定义名称都不足以稳定表达“这是谁的哪台设备”。当多人各自拥有同名设备，或已配对设备数量增长时，用户无法可靠地从设备列表和 AI/MCP 的候选列表中选择目标，容易把文件发错人。

## What Changes

- 为已配对设备增加仅保存在本机的别名；别名优先于对端设备名称和 hostname 展示，且不改变 PeerId、对端设备名称、信任级别或配对关系。
- 引入本机设备分组：用户可创建、重命名、排序和删除分组，并将一个设备加入零个或多个分组。
- 设备页提供按分组浏览、筛选和管理入口；未分组设备保持可见，在线优先的现有发送体验不变。
- MCP 的设备查询结果提供面向用户的显示名和所属分组，使 AI 在存在重名设备时先向用户确认目标，而不是以 PeerId 作为识别文本。
- 删除分组时仅移除分组关系，不取消配对、不修改设备别名或信任策略。

## Capabilities

### New Capabilities

- `device-organization`: 本机设备别名、设备分组及其持久化、管理和 MCP 展示语义。

### Modified Capabilities

- `home-device-hub`: 已配对设备需要支持别名优先展示、分组浏览和未分组设备的明确入口。

## Impact

- `src`: 设备页、发送目标选择和相关 Zustand 偏好状态增加别名/分组管理与筛选交互。
- `src-tauri`: MCP 设备列表读取本机组织元数据，并返回稳定 PeerId、显示名和分组。
- `crates/core`: 保持配对身份、对端 `OsInfo` 与信任策略的现有边界；本机组织元数据不写入 P2P 协议也不向对端同步。
- 持久化：新增可迁移的本机组织数据；旧用户默认全部设备无别名、无分组，不影响现有配对或发送行为。
