## 为什么

SwarmDrop 当前基本把设备关系处理成二元状态：已配对或未配对。收件箱存在后，SwarmDrop 可以更安全地支持低摩擦接收：先区分自有设备、协作者设备、临时设备和阻止设备，再把允许自动接收的内容送入收件箱，而不是直接散落到文件系统。

本变更引入设备信任和接收策略，作为可信数据通道的信任边界。

## 变更内容

- 新增已配对/已知设备的信任层级：
  - `owned`: 用户自己的设备
  - `collaborator`: 可信的人或设备，但默认仍需显式确认接收
  - `temporary`: 短期或分享码场景设备，带限制
  - `blocked`: 在解除阻止前不能发送或配对的设备
- 新增每设备接收策略：
  - 自动接收入站传输
  - 要求确认
  - 单次传输大小上限
  - 允许/禁止目录传输
  - 保存目的地 / 收件箱行为
  - 允许/禁止未来 MCP 向该设备发送
  - 允许/禁止 relay 路径自动接收
- 为每种信任层级增加默认策略模板。
- 配对完成后允许设置设备信任层级，并可从设备详情/操作菜单后续编辑策略。
- 在收到入站 transfer offer 时执行策略：
  - 自有设备可按策略自动接收进入收件箱
  - 协作者设备默认需要确认
  - 临时设备受 TTL 和限制约束
  - 阻止设备直接拒绝
- 与 `drop-inbox-and-transfer-activity` 集成：自动接收必须进入收件箱，策略决策应在活动与恢复中可见。
- 第一版只覆盖接收策略和设备分类；更完整的 MCP 权限 UI 可在后续基于此扩展。

## 能力

### 新增能力

- `trusted-device-policies`: 已知设备的信任层级、策略存储、策略编辑和默认模板。
- `policy-gated-receive`: 基于设备信任和接收策略决定入站 offer 的处理，包括自动接收入收件箱、策略拒绝或要求确认。

## 影响

- **crates/core**: 扩展已配对设备元数据或增加设备策略存储；暴露策略查询/更新 API；在入站 transfer offer 处理中应用策略。
- **src-tauri**: 增加读取/更新信任层级和策略的 commands/events；桥接策略自动接收/拒绝结果。
- **src**: 增加信任徽标、设备策略编辑 UI、配对后信任选择，以及反映策略决策的入站 offer 文案。
- **持久化**: 将现有已配对设备迁移为安全默认值：`collaborator`，且需要确认。
- **收件箱依赖**: 自动接收依赖收件箱目的地，不能绕过收件箱保留语义。
- **SwarmDrop-RN**: 共享策略类型可能影响移动端 bindings；移动端 UI 可在桌面行为稳定后跟进。
